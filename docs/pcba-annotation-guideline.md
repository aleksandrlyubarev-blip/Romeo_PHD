# Гайдлайн разметки PCBA для обучения VLM

> Статус: **draft v0.1** — рабочая инструкция для разметчиков и QC-инженеров.
> Связанные документы: ТЗ — [`vlm-pcba-tz.md`](./vlm-pcba-tz.md); таксономия и схема в коде —
> `Romeo_PHD/ml/pcba/taxonomy.py`, `Romeo_PHD/ml/pcba/schema.py`.
> При расхождении между этим документом и кодом — **код является источником истины**;
> расхождение фиксируется как баг и устраняется.

Цель разметки — получить точную **instance-сегментацию** всех компонентов печатной платы
в сборе (PCBA) с классами и атрибутами. Эти аннотации идут в обучение модели описания
(треки A/B/C ТЗ) и в построение эталона для вердикта good/bad.

---

## 1. Инструмент и формат

- **Формат:** COCO instance segmentation (`annotations.json` + `images/`). Поле `segmentation` —
  полигон(ы); `bbox` — `[x, y, w, h]` в пикселях; `category_id` — из таблицы §3.
- **Инструменты:** CVAT / Label Studio / Roboflow (любой с экспортом в COCO).
- **Полигоны, а не маски по пикселям:** обводим контур компонента полигоном; для прямоугольных
  корпусов достаточно 4 точек, для сложных — столько, сколько нужно для плотного прилегания.
- **Один объект = одна аннотация = один `id`.** Не объединяем несколько компонентов в один полигон.
- **Атрибуты** пишем в поле `attributes` аннотации (см. §4).
- **Изображения:** RGB, ≥ 12 Мп желательно; в `images[*]` указываем `width`, `height`,
  `board_side` (`top` / `bottom` / `unknown`).

---

## 2. Общие правила сегментации

1. **Плотное прилегание.** Контур идёт по видимой границе корпуса компонента, не захватывая
   паяльную пасту/площадку вокруг (кроме классов, где площадка и есть объект — `gold_pad_enig`,
   `test_point`).
2. **Перекрытия.** Если компонент частично перекрыт другим (радиатором, кабелем) — обводим
   **только видимую часть**. Скрытый полностью компонент не размечаем.
3. **Обрезка краем кадра.** Компонент, частично вышедший за край изображения, размечаем по
   видимой части.
4. **Минимальный размер.** Объекты мельче ~8×8 px на изображении не размечаем (помечаем сцену
   как «требует более крупного плана», см. §8). Цель — мелкие компоненты снимать крупнее.
5. **Группы одинаковых мелких компонентов** (массив пассивов) — размечаем **каждый отдельно**,
   не одной областью.
6. **Не размечаем:** дорожки/трассировку, переходные отверстия (via) без металлизации,
   тени, блики, фон вне платы.
7. **`id` аннотаций** уникальны в пределах файла; формат `id` компонента в JSON-описании —
   `c-0001`, `c-0002`, … (проставляется автоматически конвертером, разметчик нумерацией не занят).

## 3. Таксономия классов

Совпадает с `ml/pcba/taxonomy.py` (`category_id` стабильны, не переиспользуются).

| id | class | Что относится | Что НЕ относится |
|----|-------|---------------|------------------|
| 1 | `smt_passive` | SMT резисторы, конденсаторы, индуктивности, ферриты, диоды в чип-корпусе | микросхемы (→ `smt_ic`) |
| 2 | `smt_ic` | SMT-микросхемы общего назначения, память, регуляторы, драйверы, ОУ | МК/контроллеры/SoC (→ `mcu_controller`) |
| 3 | `mcu_controller` | Микроконтроллеры, контроллеры, SoC, FPGA, BMC | прочие ИС (→ `smt_ic`) |
| 4 | `tht_component` | Выводные (through-hole) компоненты: электролиты, дроссели, реле | press-fit (→ `press_fit`) |
| 5 | `press_fit` | Press-fit разъёмы и контакты (запрессованные без пайки) | паяные разъёмы (→ `connector`) |
| 6 | `connector` | Разъёмы платы, кабельные интерфейсы, гнёзда, штыревые колодки | press-fit (→ `press_fit`) |
| 7 | `screw_fastener` | Винты, гайки, стойки, клипсы, защёлки крепежа | радиаторы (→ `heatsink_mechanical`) |
| 8 | `heatsink_mechanical` | Радиаторы, экраны (shield), рамки, механические элементы | крепёж (→ `screw_fastener`) |
| 9 | `gold_pad_enig` | Позолоченные/ENIG площадки, краевые контакты (edge connector), контактные «пальцы» | обычные паяльные площадки под компонент |
| 10 | `test_point` | Тест-пойнты (контрольные точки) | `gold_pad_enig` |
| 11 | `silkscreen_marking` | Маркировка, шелкография, QR/DataMatrix, key-маркеры, обозначения позиций | травленая медь без краски |
| 12 | `pcb_bare_area` | Открытый служебный участок платы (размечается только по отдельному запросу) | — |
| 13 | `unknown` | Объект не удаётся классифицировать (требует арбитража, см. §8) | — |

**Подтипы** (конкретный корпус `0402`/`0603`, модель разъёма и т.п.) пишутся в атрибут
`subclass` (§4), а не как отдельный класс.

## 4. Атрибуты

Заполняются в `attributes` аннотации. Заполняем только те, что применимы к классу.

| Атрибут | Применим к классам | Значения | Когда заполнять |
|---------|--------------------|----------|-----------------|
| `subclass` | любой | строка (напр. `cap_0603`, `rj45`) | если корпус/модель достоверно опознаны |
| `state` | см. ниже | см. ниже | если состояние видно на фото |
| `orientation_deg` | `smt_passive`, `smt_ic`, `mcu_controller`, `tht_component`, `connector` | `0` / `90` / `180` / `270` (или точный угол) | для полярных/ключеванных компонентов |

Значения `state` (совпадают с `COMPONENT_STATE_VALUES` в коде):

- `screw_fastener`: `present` / `missing`;
- `connector`: `mated` / `unmated`, дополнительно `latched` / `unlatched`;
- `press_fit`: `seated` / `unseated`.

`orientation_deg` отсчитывается от опорного положения корпуса (ключ/pin-1/полярная метка
сверху = `0`). Если ориентация компонента не имеет значения — атрибут не заполняем.

## 5. Разметка дефектов (дефектные образцы)

На образцах с дефектами, помимо обычной сегментации компонентов, заполняем поле
`defect` аннотации затронутого компонента (или отдельную запись для `missing`/`extra_part`).
Типы — совпадают с `DEFECT_TYPES` в коде:

| Дефект | Как размечать |
|--------|---------------|
| `missing` | компонент отсутствует относительно эталона/BOM — отдельная запись с полигоном **посадочного места** и `defect=missing` |
| `extra_part` | лишний компонент — обычная аннотация + `defect=extra_part` |
| `wrong_part` | не тот компонент/класс/номинал — аннотируем фактический, `defect=wrong_part`, в `notes` ожидаемый |
| `wrong_orientation` | неверный поворот/полярность — обычная аннотация, фактический `orientation_deg` + `defect=wrong_orientation` |
| `misaligned` | сдвиг/поворот с посадочного места (без отрыва) — `defect=misaligned` |
| `damaged` | механическое повреждение (bent pins, скол, трещина) — `defect=damaged` |
| `contaminated` | загрязнение, избыток флюса, посторонние частицы — `defect=contaminated` |
| `unlatched` | разъём не защёлкнут — `state=unlatched` + `defect=unlatched` |
| `unseated` | разъём/press-fit вставлен не до конца — `state=unseated` + `defect=unseated` |

Если на образце несколько дефектов — размечаем все. Образец без единого дефекта = `OK`.

---

## 6. Эталонные образцы (golden board)

Для каждого типа изделия отбираем **заведомо хорошие** образцы (прошедшие полный QC):

- снимаем в тех же условиях (камера, ракурс, освещение), что и инспектируемые образцы;
- размечаем полно и аккуратно — это эталон, по нему строится `golden.json`
  (`ml/pcba/golden.py`) и считается diff;
- желательно ≥ 3–5 хороших образцов одного типа — допуски положения слотов считаются
  по разбросу между ними;
- эталон и его версия (тип платы, ревизия, дата) фиксируются в манифесте датасета.

---

## 7. Качество разметки

- **Гайдлайн обязателен к прочтению** перед началом; новые разметчики проходят калибровочный
  набор (10–20 изображений) с разбором ошибок.
- **Двойная разметка:** не менее 10% изображений размечаются независимо двумя людьми;
  считаем согласованность — IoU полигонов и Cohen's kappa по классам. Целевой порог
  согласованности — **уточнить** (ориентир: IoU ≥ 0.85, kappa ≥ 0.8).
- **Арбитраж:** спорные случаи и все `unknown` разбирает QC-инженер; решение дополняет §8.
- **Версионирование:** датасет и аннотации хранятся неизменяемыми снапшотами в GCS
  (манифест + хэши), как описано в §7 ТЗ.
- **Приватность:** изображения клиентов — под NDA, изолированные бакеты, контроль доступа.

---

## 8. Граничные случаи

- **Не уверены в классе** → класс `unknown`, в `notes` описать сомнение → арбитраж.
- **Компонент перекрыт > 70%** → размечаем видимую часть; если видно < ~10% — не размечаем.
- **Слипшиеся одинаковые пассивы** → стараемся разделить по видимым зазорам; если физически
  неразделимы на фото — `notes` + арбитраж.
- **Блик/пересвет скрывает компонент** → размечаем по видимому контуру; если объект полностью
  «выбит» бликом — не размечаем, помечаем сцену для пересъёмки.
- **Слишком мелко** (объекты < 8 px) → сцену помечаем как требующую крупного плана,
  такие компоненты не размечаем.
- **Маркировка поверх компонента** (`silkscreen_marking` на корпусе ИС) → размечаем обе
  аннотации отдельно (компонент и маркировку), они могут перекрываться.

---

## 9. Чек-лист разметчика перед сдачей

- [ ] Размечены все видимые компоненты, ни один не пропущен.
- [ ] Каждый компонент — отдельная аннотация, классы из таблицы §3.
- [ ] Полигоны плотно прилегают, нет захвата фона/пасты.
- [ ] Заполнены применимые атрибуты (`subclass`, `state`, `orientation_deg`).
- [ ] На дефектных образцах размечены все дефекты (§5).
- [ ] У изображения проставлены `width`, `height`, `board_side`.
- [ ] Спорные объекты помечены `unknown` + `notes`, переданы на арбитраж.
- [ ] Файл валиден как COCO и проходит конвертацию `ml/pcba/coco_to_sa2va.py` без ошибок.
