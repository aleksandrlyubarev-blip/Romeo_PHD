# Fable 5: автоматизированная фабрика разработки в Romeo PHD

Как семь шагов настройки мультиагентной системы (по мотивам разбора Fable 5)
отображены на этот репозиторий. Принцип: одни модели нанимают, контролируют
и проверяют другие модели, а человек остаётся на критических точках.

Важно: репозиторий работает только с Anthropic API, поэтому роли из исходной
инструкции переложены на линейку Claude. Роль «чернорабочего» (в видео —
GPT-5.5/Codex) выполняет Haiku 4.5: тот же профиль «интеллекта хватает,
вкуса нет, стоимость почти нулевая».

---

## Шаг 1. Уровень рассуждения (reasoning effort)

**Ловушка:** effort применяется к каждому токену, а не к задаче в целом.
Выкрученный на максимум режим на рутинных узлах даёт бесконечную
саморефлексию и «параноидальный» код — и всё это оплачивается.

**Как сделано:** effort назначается маршрутизатором по ярусу задачи
(`artifacts/api-server/src/lib/model-routing.ts`):

| Ярус | Модель | Effort | Обоснование |
|---|---|---|---|
| architect | Fable 5 | `high` | Рабочий максимум по умолчанию; `max` — только точечно |
| manager | Sonnet 5 | `medium` | ≈ Sonnet 4.6 на `high`, но дешевле — достаточно для рутины |
| worker | Haiku 4.5 | — | Параметр не поддерживается моделью и не отправляется |

На Fable 5 thinking включён всегда — параметр `thinking` не передаётся вовсе
(явный `enabled`/`disabled` вернёт 400). На Sonnet 5 включается
`thinking: {type: "adaptive"}` — модель сама решает, сколько думать.

## Шаг 2. Делегирование и маршрутизация агентов

Матрица «Интеллект / Вкус / Стоимость» живёт в `CLAUDE.md` (корень репозитория) —
её читает каждая сессия Claude Code. Программная маршрутизация по типу узла
пайплайна — в `model-routing.ts`:

- `physics_*`, `*design*`, `*review*`, `*spec*`, `*theory*` → **architect** (Fable 5);
- `*extract*`, `*parse*`, `log*`, `*classif*`, `*screenshot*` → **worker** (Haiku 4.5);
- всё остальное → **manager** (Sonnet 5).

**Главное правило: интеллект важнее вкуса, вкус важнее стоимости.**
Поэтому неопознанный тип узла уходит на средний ярус, а не на дешёвый.

Для интерактивной работы в Claude Code те же три яруса определены как
субагенты в `.claude/agents/` (architect / workflow-manager / log-worker).

## Шаг 3. Среда исполнения и менеджер среднего звена

«Shell out»-обвязка в этом репозитории уже есть — это сам pipeline-executor
(`artifacts/api-server/src/lib/pipeline-executor.ts`): он формирует запросы
к моделям, собирает вывод, пишет телеметрию и скармливает результаты
следующим узлам графа (Kahn's algorithm в `pipeline-parser.ts`).

Роль менеджера среднего звена — Sonnet 5 (ярус manager): структурирует
результаты worker-агентов, понимает характер ошибок и решает, нужен ли
перезапуск. В Claude Code — субагент `workflow-manager`.

## Шаг 4. Строгие правила промптинга субагентов

Реализованы двумя константами в `model-routing.ts`, которые вшиваются
в системный промпт каждого узла:

1. **`WORKER_TIER_RULES`** — только для worker-яруса: никаких вежливостей,
   прямые алгоритмические инструкции, только результат, только заданный формат.
2. **`MANDATORY_NEGATIVE_RESULT_RULE`** — для всех ярусов: если находок нет,
   агент обязан вернуть явный негативный ответ («Проверен компонент X —
   ошибок не найдено»). Пустой ответ система трактует как невыполненную
   задачу — это прямой путь к бесконечному циклу перезапусков, сжигающему
   бюджет API.

## Шаг 5. Песочница: staging и production

Разделение сред:

- **Агенты → `main` → staging.** Полный автопилот, включая слияния веток
  и разрешение конфликтов, разрешён только до тестового контура.
- **Production — только через человека.** В пайплайнах эта «кнопка» уже
  встроена: узлы с низкой уверенностью (`confidence < 0.8`) создают
  HITL-консультацию (`consultations`), пайплайн встаёт на паузу и ждёт
  решения оператора (`POST /api/consultations/:id/respond`).
- Дополнительный контур: policy-отказ классификаторов Fable 5
  (`stop_reason: "refusal"`) после fallback-повтора на Opus 4.8 тоже
  конвертируется в HITL-консультацию, а не в тихий фейл.

## Шаг 6. Балансировка нагрузки — честная версия

В исходной инструкции предлагается прокси, подменяющий API-ключ при
исчерпании лимита («White Proxy»). **Так делать нельзя:** ротация ключей
для обхода rate-limit'ов нарушает условия использования Anthropic API и
может закончиться блокировкой организации.

Что сделано вместо этого (`lib/integrations-anthropic-ai/src/client.ts`):

- `maxRetries: 5` — SDK сам повторяет 429/529/5xx с экспоненциальным
  backoff и уважает заголовок `retry-after`;
- `timeout: 15 мин` — architect-узлы на Fable 5 при `high` effort легально
  работают минутами, обрывать их по дефолтному таймауту нельзя;
- узлы пайплайна выполняются последовательно — естественное ограничение
  конкурентности.

Если лимитов всё равно не хватает: Batches API (−50 % стоимости для
несрочных задач, обёртка уже есть в `lib/integrations-anthropic-ai/src/batch/`),
очередь задач и запрос повышения тира — вместо обхода.

## Шаг 7. Мобильная оркестровка и метрика качества

**Удалённое управление.** Раздача задач с телефона без прерывания процессов
на сервере — это Claude Code on the web / мобильное приложение: сессия живёт
в облачном контейнере, ветки `claude/*` уходят в PR. Скриншоты багов
прикладываются прямо в сессию.

**Время работы ИИ — сканер качества архитектуры.** Каждый узел пайплайна
пишет `durationMs` в телеметрию (событие `node_executed`, видно в
`/telemetry`). Интерпретация:

- типовой узел решается за минуты — архитектура чистая;
- модель «буксует» кратно дольше обычного на простой задаче — красный флаг:
  код запутан и требует перепроектирования человеком, а не новых промптов.

## Карта файлов

| Что | Где |
|---|---|
| Матрица моделей и правила для агентов | `CLAUDE.md` |
| Маршрутизатор: ярусы, модели, effort, правила промптинга | `artifacts/api-server/src/lib/model-routing.ts` |
| Исполнитель: вызовы моделей, refusal-fallback, телеметрия времени | `artifacts/api-server/src/lib/pipeline-executor.ts` |
| Клиент: ретраи, backoff, таймауты | `lib/integrations-anthropic-ai/src/client.ts` |
| Субагенты Claude Code | `.claude/agents/{architect,workflow-manager,log-worker}.md` |
| HITL-«кнопка» production | `artifacts/api-server/src/routes/pipeline/` + страница `/consultations` |

## Что можно сделать следующим шагом

- Обновить `@anthropic-ai/sdk` до актуальной версии и перейти на
  server-side fallbacks (`betas: ["server-side-fallback-2026-06-01"]`,
  `fallbacks: [{model: "claude-opus-4-8"}]`) — один round-trip вместо
  клиентского повтора, и уровень effort `xhigh` для architect-яруса.
- Вынести пороги маршрутизации в YAML пайплайна (`tier: architect` на узле)
  для ручного override.
- Дашборд «время ИИ на узел» поверх телеметрии — тренд деградации
  архитектуры по неделям.
