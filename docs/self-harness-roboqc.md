# Self-Harness для RoboQC: обвязка, которая улучшает себя сама

Дата: 2026-07-05.
Статус: дизайн-документ + рабочий прототип цикла (`research/self-harness/`).
Первоисточник: *Self-Harness: Harnesses That Improve Themselves*,
Shanghai AI Lab, июнь 2026, arXiv:2606.09498.

## 1. Идея и мотивация

**Scaffold (каркас)** — стратегическая логика решения задачи: что и в каком
порядке думать. **Harness (обвязка)** — техническая ткань вокруг замороженной
модели: system prompts, инструменты и их схемы, управление состоянием/памятью,
политики выполнения, recovery от ошибок, verification rules, orchestration
logic. На практике bottleneck'ом agentic-системы чаще оказывается обвязка,
а не модель.

Self-Harness — паттерн, при котором агент **сам** итеративно улучшает свою
обвязку, без человека и без более сильной внешней модели. Результаты бумаги
на Terminal-Bench-2.0 (64 containerized-задачи):

| модель | до | после | прирост |
|---|---|---|---|
| MiniMax M2.5 | 40.5% | 61.9% | +21.4 п.п. (+53% rel.) |
| Qwen3.5-35B | 23.8% | 38.1% | +14.3 п.п. (+60% rel.) |
| GLM-5 | 42.9% | 57.1% | +14.2 п.п. |

Ключевое: улучшения targeted и model-specific, обобщаются на held-out задачи,
и получены с теми же локальными моделями (Qwen, GLM), на которых строится
edge-стек RoboQC.

Для RoboQC (edge-native agentic CV + decision, sub-200ms, few-shot, on-prem,
заказчик без ML-команды) это закрывает главный операционный разрыв: сейчас
каждый новый пилот (фабрика / линия / типы дефектов / освещение) требует
ручного тюнинга промптов и политик инженером. Self-Harness превращает это в
системный механизм: агент подстраивается под конкретную линию с каждой сменой.

## 2. Цикл из трёх стадий

```
        ┌──────────────────────────────────────────────────────────┐
        │                                                          │
        ▼                                                          │
┌───────────────┐    traces    ┌────────────────┐   K edits   ┌────┴─────────┐
│ 1. Weakness   │─────────────▶│ 2. Harness     │────────────▶│ 3. Proposal  │
│    Mining     │              │    Proposal    │             │  Validation  │
│ кластеризация │              │ минимальные    │             │ held-in +    │
│ ошибок по     │              │ правки, каждая │             │ held-out,    │
│ сигнатурам    │              │ привязана к    │             │ non-regres-  │
│               │              │ mined weakness │             │ sion rule    │
└───────────────┘              └────────────────┘             └──────────────┘
                                                       accept → merge → rollout
```

1. **Weakness Mining.** Прогон текущей обвязки на задачах → богатые execution
   traces → кластеризация ошибок по **сигнатуре**: причина на уровне
   верификатора (что именно провалилось: missed defect, false positive,
   format error, timeout) × механизм на уровне агента (почему: нет few-shot
   покрытия, деградация под бликом, дрейф формата вывода, бесконечный цикл)
   × контекстный бакет (тип дефекта, освещение, зона платы). Оставляются
   только recurring, actionable паттерны.

2. **Harness Proposal.** Та же модель в роли proposer'а генерирует K
   разнообразных **минимальных** правок. Ограничения: каждая правка привязана
   к конкретной mined weakness; трогает ровно одну editable surface;
   кандидаты разнообразны (не K вариаций одной идеи).

3. **Proposal Validation.** Каждый кандидат тестируется независимо против
   текущей обвязки на held-in и held-out сплитах (paired evaluation — общие
   случайные условия, чтобы сравнивать на идентичных эпизодах). Принимается
   только при улучшении хотя бы одного сплита **без деградации** другого
   (non-regression rule). Совместимые принятые правки мёржатся; цикл
   повторяется, пока не «высохнет».

## 3. Editable surfaces в RoboQC

| Surface | Что это в RoboQC | Где аналог в RomeoPHD сейчас |
|---|---|---|
| Vision prompts + few-shot | описание дефектов, примеры для VLM (Florence-2 и т.п.) по типам дефектов | персоны в `artifacts/api-server/src/lib/agent-personas.ts` |
| Decision policies | пороги pass / fail / rework / escalate, YAML/JSON-политики | YAML-пайплайны + пороги confidence в `pipeline-executor.ts` |
| Output contracts | схемы структурированного вывода VLM → PLC/MES | JSON-парсер ответа воркера в `executeNodeWithLLM` |
| Verification rules | self-check, кросс-проверка вторым ракурсом, правила верификатора | статусы RESOLVED / AMBIGUOUS / NEEDS_CLARIFICATION |
| Recovery policies | retry с подстройкой камеры/света, fallback-ветки | сейчас: fail-fast (`pipeline_failed`) — сама по себе mined weakness |
| Execution guards | step budget, таймауты, лимиты повторных инспекций | нет (editable surface для первой же итерации) |
| Memory schemas | что агент помнит о смене/партии/линии | `previousOutputs` между узлами |
| Tool schemas / handoffs | схемы инструментов, протоколы передачи между агентами | типы узлов пайплайна |

**Не** editable surfaces (заморожено): веса моделей, сам верификатор
(ground truth), код валидационного контура. Верификатор и валидатор не могут
быть объектом правок — иначе цикл учится обманывать собственный экзамен.

## 4. Таксономия failure signatures для QC

Сигнатура = `(причина верификатора, механизм агента, контекстный бакет)`.
Стартовый каталог для инспекции печатных плат:

| Причина (verifier) | Механизм (agent) | Типичный контекст | Кандидатная правка |
|---|---|---|---|
| missed_defect | no_fewshot_coverage | defect=tombstone | добавить few-shot блок для типа дефекта |
| missed_defect | glare_degradation | lighting=glare | retry-политика с подстройкой света/экспозиции |
| false_positive | connector_ambiguity | zone=connector | кросс-проверка вторым ракурсом |
| false_positive | hallucinated_defect | zone=generic | verification step (self-check) перед вердиктом |
| format_error | output_schema_drift | любой | строгая схема вывода + ретрай на невалидный JSON |
| timeout | unbounded_reinspect_loop | ambiguous frame | step budget guard |
| missed_defect | residual_classifier_miss | покрытый дефект | **не actionable** — предел модели, не обвязки |

Последняя строка принципиальна: mining обязан отличать редуцируемые ошибки
обвязки от нередуцируемого остатка модели, иначе proposer будет бесконечно
генерировать правки, которые валидация всё равно отвергнет.

## 5. Требования к traces

Weakness Mining работает ровно настолько хорошо, насколько богаты логи.
Каждый шаг агента логирует: input summary (кадр, метаданные платы, освещение),
tool calls + аргументы, outputs, результат верификатора, тайминги, ошибки.
В RomeoPHD база уже есть — `telemetry_events` (pipeline_started, node_executed,
hitl_triggered, node_error с payload). Для mining нужно расширить payload:
сейчас логируются status + confidence, но не механизм ошибки и не контекст.
Структурированные traces + embedding-индекс (Qdrant) дают кластеризацию
сигнатур и за пределами точных совпадений.

## 6. Валидация и безопасность изменений

- **Golden set (held-in):** исторические сложные дефекты + simulated factory
  сценарии; на нём майнятся слабости и меряется прирост.
- **Held-out:** свежие данные с линии, не участвующие в mining. Правка обязана
  не регрессировать здесь — это защита от оверфита на golden set.
- **Non-regression rule:** accept ⇔ (Δ ≥ ε на одном сплите) ∧ (Δ ≥ −δ на
  другом), ε ≈ 1 п.п., δ ≈ 0.5 п.п. Paired evaluation (общие эпизоды) режет
  дисперсию и позволяет меньшие ε.
- **Минимальность:** правка трогает одну surface; diff ограничен по размеру;
  правки вне списка editable surfaces отклоняются до всякой оценки.
- **Staged rollout:** shadow (кандидат считает вердикты параллельно с
  production-обвязкой, не влияя на линию) → canary (одна камера/позиция) →
  production. Плюс дрейф-контроль (embedding similarity промптов до/после)
  и human approval gate на production-стадии — в RomeoPHD это существующий
  механизм `consultations` (HITL).
- **Rollback:** обвязка версионируется как код (git); откат = revert.

## 7. Edge-топология

- **На линии (Jetson/edge):** production-обвязка + лёгкий верификатор
  (правила + маленькая модель) + сбор traces.
- **На узле фабрики (или offline):** Weakness Mining по накопленным traces,
  генерация proposals более мощной моделью, shadow-валидация на реплеях смены.
- **Rollout:** staged, с human gate. Ничего из цикла не требует облака —
  совместимо с on-prem ограничениями заказчиков.

## 8. Что реализовано в прототипе

`research/self-harness/self_harness_poc.py` — рабочая реализация всех трёх
стадий на синтетическом, но механистически честном QC-окружении (stdlib-only,
детерминирован по seed). «Агент» — стохастический симулятор, чьи failure modes
управляются surfaces обвязки, поэтому прототип демонстрирует **механику цикла**,
а не способности конкретной LLM (LLM-proposer подключается вместо каталога
правок без изменения контура).

Результат эталонного прогона (seed 42, 1200 задач, сплит 50/50):

| | held-in | held-out |
|---|---|---|
| baseline (минимальная обвязка) | 56.3% | 53.3% |
| после 1 раунда цикла | 92.7% | 91.5% |
| прирост | **+36.3 п.п.** | **+38.2 п.п.** |

Свойства, которые прогон демонстрирует и которые обязаны сохраниться в
production-версии:

1. Mining находит все 9 заложенных recurring-сигнатур и ранжирует по частоте.
2. Все 8 таргетных минимальных правок проходят валидацию (улучшение на обоих
   сплитах, без регрессий).
3. Оба «дистрактора» отклонены: `paranoid_reject_everything` (поднимает recall,
   но рушит false-positive rate — ловится non-regression rule) и
   `add_generic_caution_instruction` («будь внимательнее» — нулевой эффект).
4. Цикл сам останавливается («высыхает»), когда остаются только
   `residual_classifier_miss` — нередуцируемый остаток модели, к которому
   у каталога нет actionable-правок.
5. Прирост на held-out не меньше, чем на held-in, — правки обобщаются, а не
   оверфитятся на сплит, где майнились.

Числа прироста — свойство синтетического окружения и иллюстративны;
переносимая часть — контур mining → proposal → validation и его инварианты.

## 9. Roadmap

1. **Traces:** расширить payload `telemetry_events` до полной сигнатуры
   (механизм + контекстный бакет); добавить реплей эпизодов.
2. **Первый модуль:** decision loop по дефектам (или multi-step inspection
   агент) — один модуль, не вся система.
3. **LLM-proposer:** заменить каталог правок на frozen-LLM-proposer над
   реальными файлами обвязки (промпты, YAML-политики) с diff-ограничениями
   из §6.
4. **Golden set:** собрать held-in (исторические сложные кейсы + симуляция)
   и held-out (свежие данные) сплиты; завести paired evaluation.
5. **Rollout-контур:** shadow-режим в OpenClaw, canary на одной позиции,
   human gate через существующий HITL.
6. **Расширение:** тот же цикл для оркестрационного слоя (tool schemas,
   handoff protocols, context management) и исследовательских агентов
   (Andrew Swarm).
