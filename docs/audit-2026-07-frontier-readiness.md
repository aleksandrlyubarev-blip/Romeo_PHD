# Аудит фабрики: готовность к фронтир-моделям и RWDM-бенчмарку

Дата: 2026-07-07. Метод: три параллельных аудит-агента (маршрутизация моделей,
RWDM-бенчмарк, инфраструктура исполнения) + сверка с правилами `CLAUDE.md`.

## Резюме

Фабрика соответствует собственным правилам в части ярусов, effort и fallback,
но **не готова** к двум заявленным целям:

1. **Мультипровайдерность отсутствует.** Реализован только Anthropic;
   упоминания OpenAI/Gemini — мёртвый шаблонный код. Подписки на другие
   американские фронтир-модели (OpenAI, Google, xAI) использовать нечем.
2. **Участие агентов в RWDM-бенчмарке — фикция на уровне кода.** Пайплайн
   `physics-engineer-rwdm` — чисто текстовый LLM-диалог: он не запускает
   `rwdm_qbench_prototype.py`, не читает `results/summary.json` и не может
   проверить ни одно число. Документ, прототип и агент существуют раздельно.

Плюс найден один критический баг в HITL и несколько расхождений кода с
`CLAUDE.md`.

## Критические находки (P0)

### 1. HITL-гейт не гранулярен по узлам — баг

- `artifacts/api-server/src/routes/pipeline/index.ts:349-360` — ответ на
  консультацию обновляет статус **всех** узлов пайплайна
  (`where eq(pipelineNodes.pipelineId, ...)`), а не только узла консультации;
  `consultation.nodeId` не используется.
- `routes/pipeline/index.ts:208-270` — `POST /pipelines/:id/resume` принимает
  `nodeId` в теле (`ResumePipelineBody`), но полностью игнорирует его: выборка
  и апдейт фильтруются только по `pipelineId`.
- Следствие: одобрение одной консультации «протаскивает» статус на посторонние
  (в т.ч. завершённые) узлы. Это ломает главную гарантию безопасности фабрики —
  точечное человеческое подтверждение.

### 2. Пустой ответ субагента принимается как успех

`MANDATORY_NEGATIVE_RESULT_RULE` (`model-routing.ts:130-139`) — только текст в
промпте. На сервере (`pipeline-executor.ts:160-179`) нет проверки `output` на
пустоту/заглушку: пустая строка или "N/A" фиксируется как `RESOLVED`.
Проверяется лишь `stop_reason === "refusal"`. Правило из `CLAUDE.md` о
трактовке пустого ответа как невыполненной задачи не enforce-ится.

### 3. Physics-контур не связан с кодом бенчмарка

- Узлы `physics_*` исполняются как один LLM-вызов (`pipeline-executor.ts`),
  без tool-use, без запуска кода, без чтения результатов.
- Пресет `physics-engineer-rwdm` (`artifacts/romeo-phd/src/lib/buxter.ts:385-427`)
  — 6 текстовых узлов без единой точки контакта с
  `research/quantum-benchmark/`.
- Автономное развитие бенчмарка агентами при такой архитектуре невозможно:
  нет цикла «агент → запуск кода → численная проверка → вердикт».

## Мультипровайдерность: что нужно для «всех американских фронтир-моделей»

Единственная точка привязки к Anthropic — `callModelWithRoute`
(`pipeline-executor.ts:47-81`) с прямым `anthropic.messages.create`. План:

1. `ModelRoute` (`model-routing.ts:25`) — добавить
   `provider: "anthropic" | "openai" | "google" | "xai"`; Anthropic-специфику
   (`effort`, `thinking`, `useRefusalFallback` через `stop_reason: "refusal"`)
   вынести в `anthropicOptions`, аналоги — в провайдер-специфичные опции.
2. Провайдер-агностичный интерфейс `LLMClient.complete(route, system, user)`
   с адаптерами по образцу `lib/integrations-anthropic-ai/src/client.ts`:
   `lib/integrations-openai-ai`, `lib/integrations-google-ai`,
   `lib/integrations-xai-ai` (env-ключи, retry в SDK, единый формат ответа).
3. Матрицу `MODELS` расширить кросс-провайдерными ролями. Разумная стартовая
   раскладка: architect остаётся Fable 5 (fallback Opus 4.8), но для
   **кросс-проверки физики** добавить второе мнение от модели другого
   провайдера (независимый вывод формул — лучший детектор галлюцинаций);
   worker-ярус можно разгружать дешёвыми не-Anthropic моделями при исчерпании
   лимитов — это легальная альтернатива запрещённой ротации ключей.
4. Удалить вводящий в заблуждение мёртвый код (см. «Мусор» ниже), чтобы поиск
   по репо не создавал впечатление уже сделанной интеграции.

## RWDM-бенчмарк: находки по `rwdm_qbench_prototype.py`

Научные:

- **`rate_scale` (строки 156-164) — подгонка.** Перебор 0.5/1.0/2.0 вместо
  вывода из `w_ij = b_ij²·g(Δω)` нарушает политику §8 дизайн-документа
  («никакой подгоночной экспоненты») по духу: масштаб скорости получен
  перебором, а не расчётом.
- **Шкалы τ квантовой и классической веток не откалиброваны**, но рисуются на
  одной оси (Tier A, строки 236-243): `τ_q = B0·t` против классической шкалы
  `~B0²·τ_c·t`. Риск признан в §11 документа, но на графике не помечен.
- Верификационные гейты §5 (парный бенчмарк τ≤0.5, моменты M₂/M₄, регулярный
  предел c=1, `ε(N,τ)≤0.05`) не реализованы ни одной строкой кода.
- Малое `ng=3` для Tier B: полуребро суперячейки ~1.5a — конечноразмерные
  артефакты от размера ячейки не отделены от артефактов малого N.

Инженерные:

- Ни одного `assert`/теста: нет проверок эрмитовости H, нулевых сумм столбцов
  генератора, нормировки P₀₀(0)=1 — хотя `docs/physics-engineer-agent.md`
  называет их обязательными.
- Tier B — плотные матрицы `2^n × 2^n` без блочности по полному Sz (который
  гамильтониан сохраняет): дорожная карта N=20-30 на этом коде невыполнима
  (2²⁰ — ~8 ТБ на матрицу). Нужен sparse/блочный ED.
- Все параметры захардкожены в `main()` — каноническую сюиту QB-core/bridge/
  check (§4) без правки кода не воспроизвести. Нужен CLI
  `(ng, c, orientation, field_case, seed)`.
- В `summary.json` нет git hash / версии кода — результаты в `results/`
  невозможно связать с породившим их кодом.
- Плюс: сиды детерминированы, линейная алгебра (eigh, автокоррелятор через
  матричные элементы) реализована корректно.

## Приоритезированный план

**P0 — безопасность и честность фабрики**
1. Починить HITL: фильтровать апдейты по `consultation.nodeId`; использовать
   `nodeId` из `ResumePipelineBody`.
2. Серверная валидация output: пустой/заглушечный ответ → не `RESOLVED`,
   а повтор или консультация.
3. Tool-узел в pipeline-executor: запуск скрипта + чтение артефактов, чтобы
   physics-пайплайн реально исполнял и проверял код бенчмарка.

**P1 — фронтир-модели и экономика (актуально при 14% лимита Fable)**
4. Провайдер-адаптер + интеграции OpenAI/Google/xAI (см. план выше).
5. Prompt caching (`cache_control` на system-блоке) — бесплатная экономия на
   каждом узле.
6. Реальный Batches API для worker-яруса (−50% стоимости оффлайн-задач).
7. Синхронизировать `.claude/agents/architect.md` с матрицей (Fable 5) либо
   задокументировать, что интерактивный архитектор сознательно на Opus.

**P2 — бенчмарк и наблюдаемость**
8. Тесты и инварианты прототипа (assert + pytest + CI), гейты §5 в коде.
9. Sparse/Sz-блочный ED, CLI-параметризация, git hash в `summary.json`.
10. Вывод `rate_scale` из физики вместо перебора; калибровка шкал τ.
11. Агрегация `durationMs` (пороги, алерты) — довести заявленную метрику
    качества архитектуры до рабочего состояния.
12. Удалить мёртвый код; настроить branch protection для гейта
    staging→production.
